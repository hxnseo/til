# Binary Search Tree — 핵심 정리

> 복습용 개념 정리. 코드 구현은 `BST.java` 참고.

---

## 0. 전제: Binary Tree & Traversal

- **Binary tree**: 모든 노드가 자식 ≤ 2개인 ordered tree (왼/오 구분됨).
- **3가지 순회 (재귀)**:
  - **Preorder**: 루트 → 왼쪽 → 오른쪽
  - **Inorder**: 왼쪽 → 루트 → 오른쪽
  - **Postorder**: 왼쪽 → 오른쪽 → 루트
- 수식 트리 `(A+B)*(C+(D+E))` 예: postorder 결과 = **후위 표기법(postfix)**.
  → *후위표기법 = 수식트리를 postorder 순회한 결과.*

---

## 1. BST 정의

BST는 다음 둘 중 하나:
- 빈 트리, 또는
- 다음을 만족하는 binary tree:
  - 루트가 key를 가짐
  - **왼쪽 서브트리의 모든 key < 루트 key < 오른쪽 서브트리의 모든 key**
  - 양쪽 서브트리도 각각 BST
- **key는 서로 다름 (distinct, 중복 없음)**

> 직관: 정렬 배열에서 binary search를 할 때 "매 단계의 비교 결정"을 그대로 트리로 그리면 BST가 됨.

---

## 2. 왜 BST를 쓰나 (vs 정렬 배열)

| | 정렬 배열 | BST |
|---|---|---|
| search | O(log n) | **O(h)** |
| insert | O(n) (뒤 원소 밀기) | **O(h)** |
| delete | O(n) | **O(h)** |

- 정렬 배열은 search는 빠르지만 삽입/삭제에서 원소 이동 때문에 O(n).
- BST는 셋 다 **트리 높이 h에 비례**. h만 작으면 전부 빠름.

---

## 3. 기본 연산 (전부 O(h))

### Search
- 매 노드에서 key 비교 → 작으면 왼쪽, 크면 오른쪽. null 만나면 없음.
- *단순 트리 순회(모든 노드 방문)와 달리 방향을 정해 내려가므로 빠름.*

### Insert
- 들어갈 자리를 탐색 → **리프로 덧붙이기**.
- 비용 = 탐색 O(h) + 리프 부착 O(1) = **O(h)**.
- 이미 있는 key면 (distinct 규칙) 삽입 무시.

### Delete — 3가지 경우 ★
1. **리프**: 그냥 떼어냄.
2. **자식 1개**: 그 자식 서브트리를 삭제 노드 자리에 그대로 끌어올려 재부착.
   - 서브트리 값들은 원래 그 위치의 대소 관계를 그대로 유지하므로 BST 속성 안 깨짐.
3. **자식 2개**: **왼쪽 서브트리의 최댓값(predecessor)** 또는 **오른쪽 서브트리의 최솟값(successor)** 을 삭제 노드 자리로 끌어올리고, 그 노드를 대신 삭제.
   - *왜 이 값?* 오른쪽 최솟값조차도 여전히 (왼쪽 전부보다 크고) (오른쪽 나머지 전부보다 작음) → BST 속성 완벽 유지.
   - *대체 노드는 자식이 최대 1개*: 최솟값을 찾으러 왼쪽 끝까지 갔으므로 왼쪽 자식이 없음 → 1개-자식 케이스로 환원.
   - **장점**: 서브트리를 통째로 아래에 매다는 방식과 달리 **트리 높이가 안 커짐**.

**최댓값/최솟값 찾는 법:**
- 왼쪽 서브트리 최댓값: 왼쪽 자식으로 한 번 → 그 다음 오른쪽으로 끝까지.
- 오른쪽 서브트리 최솟값: 오른쪽 자식으로 한 번 → 그 다음 왼쪽으로 끝까지.

---

## 4. ★ 핵심 성질: Inorder = 정렬

BST에 **inorder 순회**를 하면 모든 key가 **오름차순**으로 나옴.
- 이유: (왼<루트<오른) 정의 + (왼→루트→오른) 순회 순서가 결합 → 항상 작은 것부터 방문.

---

## 5. Join (합치기)

- **입력**: BST 두 개 A, B. **전제: A의 모든 key < B의 모든 key.**
- **출력**: A∪B를 담은 하나의 BST (A, B는 파괴됨).
- **방법**: A의 **최댓값**(가장 오른쪽)을 새 루트로 올리고, A의 나머지를 왼쪽 서브트리, B 전체를 오른쪽 서브트리로 부착.
  - 두 자식 삭제에서 successor 빼내던 원리와 사실상 동일.
- **시간**: O(h).

---

## 6. Split (분할)

- **입력**: BST 하나 + 기준 key x.
- **출력**:
  1. x가 트리에 있는지 여부 (found)
  2. **less** 트리 (x보다 작은 모든 원소) + **greater** 트리 (x보다 큰 모든 원소)
- **노드 수 관계**:
  - x가 **없으면**: 원본 노드 수 = less + greater 노드 수 (아무것도 안 버려짐).
  - x가 **있으면**: 그 노드는 어느 쪽에도 안 들어가 **버려짐** → 원본 = less + greater + 1.
- **방법**: 루트에서 내려가며 각 노드를 less/greater의 "열린 슬롯"에 부착.
  - cur.key > x → cur와 오른쪽 서브트리는 greater 행, 왼쪽으로 계속 탐색.
  - cur.key < x → 대칭 (less 행).
- **시간**: O(h). **결과 트리 높이 ≤ 원본 높이** (재부착이 항상 아래 방향으로만 일어나므로).

---

## 7. Time Complexity 정리 ★

- search / insert / delete 전부 **O(h)** (h = 트리 높이). ← O(log n) 아님!
- **h가 얼마나 커질 수 있나?**
  - 삽입 순서가 **uniform random permutation**이면 → **E[h] = O(log n)** → 전부 O(log n).
  - **worst case (정렬된 순서로 삽입)** → 한쪽으로만 늘어진 사슬 → **h = O(n)** → search·insert·delete 전부 O(n). 정렬 배열보다 나을 게 없음.
- worst case를 막으려면 → 균형을 강제로 유지하는 **balanced tree (AVL, red-black 등)** 필요 → h = O(log n) 보장.

---

## 8. 내부 표현: 반드시 Pointer

- **pointer(참조) 표현**: 노드마다 left/right 참조를 직접 가짐. 리프는 null.
- **왜 array 표현을 안 쓰나?**
  - 삭제 시 서브트리를 재부착해야 하는데, array면 서브트리의 **모든 노드 데이터를 새 인덱스로 일일이 이동** → 서브트리 크기에 비례하는 시간(끔찍).
  - pointer면 참조 몇 개만 바꾸면 끝 → 재부착 O(1).
  - ⇒ 앞서 말한 O(h) 수행시간은 **pointer 표현 전제** 하의 결과.

---

## 9. Iterator (Pre / In / Post)

- **이터레이터 vs 순회**: 순회는 "어떤 순서로 방문하나"라는 전략. 이터레이터는 그걸 `hasNext()`/`next()`(여기선 `atEnd()`/`getData()`/`next()`)로 한 칸씩 꺼내 쓰게 포장한 객체.
- **왜 직접 만든 Stack이 필요한가?**
  - 기존 재귀 순회는 한 번 호출되면 **중간에 멈춰서 한 노드만 반환하는 게 불가능** (콜스택에 쌓여 끝까지 돌아감).
  - 그래서 컴퓨터가 재귀 실행 시 쓰는 **콜스택을 명시적 Stack으로 직접 시뮬레이션**. 스택엔 값이 아니라 **Node 참조**를 push/pop.
- **각 이터레이터 초기화/진행:**
  - **PreIter**: 생성 시 루트만 push. `next()` = pop 후 (오른쪽 먼저, 왼쪽 나중에) push → LIFO라 왼쪽이 먼저 나옴.
  - **InIter**: 생성 시 왼쪽 경로 끝까지 push(`pushLeft`). `next()` = pop 후, 그 노드의 오른쪽 자식에 대해 다시 `pushLeft`.
  - **PostIter** (가장 까다로움): 생성 시 갈 수 있는 데까지 왼쪽(없으면 오른쪽)으로 내려가며 push(`pushNextDesc`). `next()` = pop 후, 방금 팝한 게 부모의 **왼쪽 자식**이었으면 이제 부모의 **오른쪽 서브트리**를 `pushNextDesc`.

---

### 한 줄 요약
> BST = "정렬 배열의 검색 속도(O(log n))는 유지하면서 삽입/삭제까지 O(h)로 빠르게" 만든 구조. 단, h가 작아야(=균형) 이득. inorder는 항상 정렬. 삭제 2자식 케이스·split·join이 손코딩 핵심. 반드시 pointer 표현, 이터레이터는 명시적 stack으로 재귀를 흉내.
