# CH11. JavaScript Events

> **한 줄**: 사용자의 동작(클릭·키·마우스·폼)에 JS 함수를 "연결(등록)"해서 페이지를 동적으로 바꾸는 것.

## 1. 이벤트 핸들러 등록 3가지 방식 (★시험 단골)

| 방식 | 형태 | 특징 |
|---|---|---|
| **Inline model** | `<element onmousedown="script">` | HTML 속성으로 처리. HTML에 직접 박음 |
| **Traditional model** | `object.onmousedown = function(){...}` | DOM 노드의 이벤트 *프로퍼티*에 함수 대입. 하나만 가능 |
| **addEventListener** | `object.addEventListener("mousedown", script)` | 가장 권장. **같은 타입 핸들러 여러 개** 등록 가능 |

> 핵심 차이: traditional은 `onclick`에 함수 **하나**만 덮어쓰지만, `addEventListener`는 **여러 개** 붙일 수 있음.

## 2. addEventListener() 파라미터

```js
object.addEventListener(event, function, useCapture);
```

| 파라미터 | 설명 |
|---|---|
| `event` | 필수. 이벤트 이름. **"on" 접두사 빼기** → `"click"` (O), `"onclick"` (X) |
| `function` | 필수. 실행할 콜백 함수 |
| `useCapture` | 선택. Boolean. `true`=캡처링 단계 실행, `false`(기본)=버블링 단계 실행 |

- `removeEventListener()`: addEventListener와 **똑같은 인자**를 넘겨서 핸들러 제거.
- **이벤트 핸들러 = 콜백(callback)**. 이벤트에 핸들러를 거는 행위 = "registering(등록)".

## 3. Event 객체 (모든 이벤트의 부모)

모든 이벤트 객체는 `Event` 객체 기반 → MouseEvent, KeyboardEvent도 Event의 속성/메서드 사용 가능.

| 속성 | 설명 |
|---|---|
| `target` | 이벤트를 발생시킨 요소 반환 (read-only) ★ |
| `type` | 이벤트 이름 반환 |
| `bubbles` | 버블링 이벤트인지 (true/false, read-only) |
| `cancelBubble` | 버블링을 위로 전파할지 설정/반환 |
| `timeStamp` | 문서 로딩~이벤트 생성까지 밀리초 |

## 4. Event Bubbling (이벤트 버블링) ★중요

- **정의**: 자식 요소에서 발생한 이벤트가 부모 요소로 "거품처럼 올라가는(bubble up)" 과정.
- 이벤트는 먼저 그 요소의 핸들러 → 그다음 부모 요소의 핸들러 순으로 전달됨.
- 자식에서만 처리하고 싶으면 → 자식 핸들러 코드에서 **`event.cancelBubble`** 로 버블링을 막아야 함.

## 5. MouseEvent

| 속성 | 설명 |
|---|---|
| `altKey` / `ctrlKey` / `shiftKey` | 해당 키가 눌린 상태였는지 (true/false) |
| `button` | 눌린 버튼: **0=왼쪽, 1=가운데, 2=오른쪽** |
| `which` | 눌린 버튼: **0=없음, 1=왼쪽, 2=가운데, 3=오른쪽** (button과 숫자 다름! ★함정) |
| `clientX` / `clientY` | 현재 윈도우 기준 좌표 |
| `screenX` / `screenY` | 스크린 기준 좌표 |

**마우스 이벤트**: `onclick`, `ondblclick`, `onmousemove`, `onmouseover`, `onmouseout`, `onmousedown`, `onmouseup`
- **발생 순서**: `onmousedown` → `onmouseup` → `onclick` (★시험)

## 6. KeyboardEvent

| 속성 | 설명 |
|---|---|
| `altKey`/`ctrlKey`/`shiftKey` | 해당 키 눌림 여부 |
| `charCode` | onkeypress를 발생시킨 키의 유니코드 문자 코드 |
| `key` | 이벤트가 나타내는 키의 값 |

**키보드 이벤트**: `onkeydown`, `onkeypress`, `onkeyup`
- **발생 순서**: `onkeydown` → `onkeypress` → `onkeyup` (★시험)
- `onkeypress`는 **모든 키에서 발생 안 함** (ALT, CTRL, SHIFT, ESC 등 제외)
- → **모든 키 감지하려면 `onkeydown` 사용** (모든 키에서 동작하니까)

## 7. Form 이벤트

| 이벤트 | 발생 시점 |
|---|---|
| `onchange` | 폼 요소 내용/선택/체크 상태가 바뀔 때 |
| `onblur` | 요소가 포커스를 **잃을** 때 |
| `onfocus` | 요소가 포커스를 **얻을** 때 (클릭/Tab 이동) |
| `onreset` | 폼이 리셋될 때 |
| `onsubmit` | 폼이 제출될 때 |

## 8. 🔑 손코딩 핵심 코드

**(1) 폼 submit → 입력값 alert** — 백틱(템플릿 리터럴) 주의:
```js
document.addEventListener('DOMContentLoaded', function() {
  document.querySelector('form').onsubmit = function() {
    const name = document.querySelector('#name').value;
    alert(`Hello, ${name}!`);   // 따옴표 아닌 백틱 ` `
  };
});
```

**(2) select 변경 시 색 바꾸기** — `this` = 이벤트 받은 객체:
```js
document.querySelector('select').onchange = function() {
  document.querySelector('#hello').style.color = this.value;
};
```

**(3) load 이벤트 + 타이머** (몇 초 봤는지):
```js
let seconds = 0;
function startTimer() { window.setInterval(updateTime, 1000); }
function updateTime() {
  ++seconds;
  document.getElementById("soFar").innerHTML = seconds;
}
window.addEventListener("load", startTimer, false);
```

**(4) 🔑🔑 스크롤이 바닥에 닿았는지 감지** — CH13 무한스크롤에서 재등장. **꼭 외워**:
```js
window.onscroll = () => {
  if (window.innerHeight + window.scrollY >= document.body.offsetHeight) {
    // 페이지 끝에 도달!
  }
};
```
- `window.scrollY`: 아래로 스크롤한 픽셀 (맨 위면 0)
- `document.body.offsetHeight`: 페이지 전체 높이
- `window.innerHeight`: 보이는 창 높이

# CH12. AJAX

> **한 줄**: `XMLHttpRequest`로 페이지 **전체 새로고침 없이** 서버와 데이터를 비동기로 주고받아 페이지 일부만 갱신하는 기법.

## 1. AJAX 개념

- **AJAX** = **A**synchronous **JA**vaScript and **X**ML. 2005년 Jesse James Garrett가 만든 용어.
- 그 자체가 기술이 아니라 **여러 기술을 묶은 용어**. 핵심은 `XMLHttpRequest`.
- 백그라운드에서 데이터 교환 → **전체 페이지 안 다시 불러오고** 일부만 업데이트 → 앱이 빠르고 반응성 좋아짐.

## 2. 동기 vs 비동기 웹앱 ★

| | **동기 (Synchronous)** | **비동기 (Asynchronous, AJAX)** |
|---|---|---|
| 요청 처리 | 클라가 서버 응답을 **기다림** | 처리 중에도 **사용자가 계속 조작 가능** |
| 페이지 | **전체** 다시 로드, 잠깐 화면 빈칸 | **부분 갱신(partial page update)** |
| 응답 시 | 새 페이지 렌더 | XMLHttpRequest가 **콜백 함수** 호출 |

## 3. 🔑 XMLHttpRequest 기본 패턴 (손코딩 핵심)

```js
// Step 1: 요청 만들기
httpRequest = new XMLHttpRequest();
httpRequest.onreadystatechange = displayContents;  // 응답 처리할 함수 지정
httpRequest.open('GET', 'https://.../users/1');     // 요청 명시
httpRequest.send();                                 // 실제 전송

// Step 2: 응답 처리
function displayContents() {
  if (httpRequest.readyState === XMLHttpRequest.DONE) {  // 끝났는지(4)
    if (httpRequest.status === 200) {                    // 성공인지
      var myObj = JSON.parse(httpRequest.responseText);
      document.getElementById("demo").innerHTML = 'Name: ' + myObj.name;
    } else {
      alert('There was a problem with the request.');
    }
  }
}
```

## 4. XMLHttpRequest 메서드

| 메서드 | 설명 |
|---|---|
| `new XMLHttpRequest()` | 객체 생성 |
| `open(method, url, async, user, psw)` | 요청 명시. method=GET/POST, async=true(비동기)/false(동기) |
| `send()` | 서버로 전송 (GET용) |
| `send(string)` | 전송 (POST용, 데이터 같이) |
| `setRequestHeader()` | 헤더에 label/value 추가 |
| `abort()` | 현재 요청 취소 |

## 5. XMLHttpRequest 속성 ★ (readyState 숫자 외우기)

| 속성 | 설명 |
|---|---|
| `onreadystatechange` | readyState 바뀔 때 호출될 함수. **총 4번** 트리거(1~4) |
| `readyState` | **0**: 미초기화 / **1**: 서버연결됨 / **2**: 요청받음 / **3**: 처리중 / **4**: 완료&응답준비 |
| `responseText` | 응답을 **문자열**로 |
| `responseXML` | 응답을 **XML**로 |
| `status` | 상태 번호. **200=OK**, 403=Forbidden, 404=Not Found |
| `statusText` | 상태 텍스트 ("OK", "Not Found") |

## 6. 응답 처리 순서 (Step 2 정리)

1. 상태 확인: `if (httpRequest.readyState === XMLHttpRequest.DONE)`
2. HTTP 상태코드 확인: `if (httpRequest.status === 200)`
3. 데이터 사용: `responseText`(문자열) 또는 `responseXML`(XMLDocument)

## 7. jQuery AJAX (참고 — 형태만)

```js
$(selector).load(URL, Data, Callback);   // 서버 데이터를 선택 요소에 넣음
$.get(URL, callback);                    // GET 요청
$.post(URL, data, callback);             // POST 요청
```

# CH13. Promise ⭐⭐⭐ (이 챕터가 시험 비중 큼)

> **한 줄**: 비동기 작업의 **미래의 성공/실패를 담는 객체**. `.then()`으로 체이닝, `.catch()`로 에러 한곳 처리. 콜백 지옥 해결사.

## 1. 동기 vs 비동기 + 왜 Promise?

- **동기**: 한 번에 한 작업만. 오래 걸리는 함수(소수 생성 등) 돌리면 **화면 멈춤(unresponsive)** — 입력도 안 됨.
- **비동기**: 여러 작업 동시 진행. 서버에서 리소스 가져오기 같은 **blocking 작업**을 효율적으로.
- 옛날엔 **콜백(callback)** 으로 비동기 구현 → 중첩되면 **"pyramid of doom" (콜백 지옥)** → 읽기·디버깅 어려움.
- 현대 API는 대부분 **Promise 기반**.

```js
// 콜백 지옥 예 (피자 주문) — 읽기 끔찍
setTimeout(function() {
  requestMenu();
  setTimeout(function() {
    chooseTopping();
    setTimeout(function() { /* ... 계속 중첩 ... */ }, ...);
  }, ...);
}, ...);

// Promise 체인 — 깔끔
orderPizza()
  .then(placeOrder)
  .then(eatPizza)
  .then(payForPizza)
  .catch(badPizza);   // 에러는 여기 한 곳에서!
```

## 2. Promise 정의 & 3가지 상태 ★★ (반드시 암기)

- **정의**: 비동기 함수가 **반환하는 객체**. 비동기 작업의 **최종 완료(eventual completion) 또는 실패**를 나타냄.
- "지금은 답 못 주지만, 가능한 빨리 답 줄게" — 언제·성공할지 불확실.
- Promise를 반환할 때 보통 작업은 **아직 안 끝남**. 하지만 성공/실패를 처리할 **메서드를 제공**.

| 상태 | 의미 | 호출되는 핸들러 |
|---|---|---|
| **pending** | 생성됨, 아직 성공·실패 안 함 (작업 진행 중) | — |
| **fulfilled** | 비동기 함수 **성공** | `then()` 핸들러 |
| **rejected** | 비동기 함수 **실패** | `catch()` 핸들러 |

## 3. then / catch / finally ★

- **`then()`**: 성공 시 실행될 핸들러 포함. 입력으로 작업 **결과(Response 객체 등)** 받음. **또 다른 Promise를 즉시 반환** → 그래서 **체이닝** 가능 → 여러 비동기를 **순서대로** 실행.
- **`catch()`**: 실패 시 실행. **모든 에러를 한 곳**에서 처리 (최상위 1번만).
- **`finally()`**: 성공/실패 **상관없이** 마지막에 무조건 실행.

```js
myPromise
  .then(response => { doSomething(response); })
  .catch(e => { returnError(e); })
  .finally(() => { runFinalCode(); });   // 항상 실행
```

## 4. 🔑 fetch + Promise 기본 패턴 (손코딩 1순위)

```js
fetch('products.json')
  .then(function(response) {
    return response.json();      // JSON 데이터 반환 (이것도 Promise 반환)
  })
  .then(function(json) {
    let products = json;
    initialize(products);
  })
  .catch(function(err) {
    console.log('Fetch problem: ' + err.message);
  });
```

- `fetch()`: 브라우저가 서버에 HTTP 요청. **XMLHttpRequest의 더 간단/효율 버전**. **Promise 객체를 즉시 반환**.

## 5. 비동기 코드의 본질 (실행 순서 함정) ★★

```js
console.log('Starting');
let image;
fetch('coffee.jpg').then((response) => {
  console.log('It worked :)');
  return response.blob();        // Binary Large Object (이미지/사운드/비디오)
}).then((myBlob) => {
  let objectURL = URL.createObjectURL(myBlob);
  image = document.createElement('img');
  image.src = objectURL;
  document.body.appendChild(image);
}).catch((error) => { console.log(error.message); });
console.log('All done!');
```

**출력 순서**: `Starting` → **`All done!`** → `It worked :)`
- 이유: `fetch().then()`은 **비동기** → 나중에 실행됨. 그 사이 동기 코드(`All done!`)가 먼저 끝남.
- → `console.log('All done! ' + image.src)`를 밖에서 하면 **`image is undefined` 에러** (아직 안 만들어짐).
- **해결**: `image.src` 쓰는 부분도 `.then()` 안으로 넣어서 전부 비동기 체인으로.

## 6. Promise vs Event Listener 차이 ★

- Promise는 **딱 한 번만** 성공 또는 실패. 두 번 안 됨.
- 이미 성공/실패한 **후에** 핸들러를 **나중에 추가**해도 → **올바른 핸들러가 호출됨** (이미 resolve됐어도).

## 7. Promise 조합 (composition)

- **`Promise.all([a, b, c])`**: 배열의 **모든** Promise가 fulfill되면 → `.then()`에 **모든 응답 배열** 전달.
  - 하나라도 reject되면 → **전체가 즉시 reject**.
- **`Promise.any()`**: **하나라도** fulfill되면 됨.

```js
Promise.all([a, b, c]).then(values => { /* values = 응답 배열 */ });
```

## 8. async / await (Promise 문법 설탕) ★★

- `async`: 함수가 **Promise를 반환**하게 만듦. (값을 직접 반환하지 않고 Promise로 감쌈)
- `await`: async 함수 **안에서만** 사용. Promise 기반 함수 앞에 붙여 **그 줄에서 코드를 멈추고(pause)** Promise가 settle될 때까지 대기. fulfill값 반환, reject값은 throw.
- **에러 처리는 `try/catch`** 사용 (`.then/.catch` 아님!). ★시험 함정

```js
async function hello() { return "Hello"; }
hello();   // 값이 아니라 Promise를 반환!
hello().then((value) => console.log(value));  // 값 꺼내려면 then
```

**🔑 async/await로 fetch 다시 쓰기** (손코딩):
```js
async function myFetch() {
  try {
    let response = await fetch('coffee.jpg');     // Promise 대신 Response 객체 바로 받음
    if (!response.ok) {
      throw new Error(`HTTP error: ${response.status}`);
    }
    let myBlob = await response.blob();
    let objectURL = URL.createObjectURL(myBlob);
    let image = document.createElement('img');
    image.src = objectURL;
    document.body.appendChild(image);
  } catch (error) {
    console.log('There has been a problem: ' + error);
  }
}
myFetch();
```

## 9. 🔑 무한 스크롤 (Infinite Scroll) — CH11 스크롤감지 + fetch 결합

```js
let counter = 1;
document.addEventListener('DOMContentLoaded', load);

window.onscroll = () => {
  if (window.innerHeight + window.scrollY >= document.body.offsetHeight) {
    load();   // 바닥 닿으면 더 불러옴
  }
};

function load() {
  const start = counter;
  const end = start + 1;
  counter = end + 1;
  fetch(`https://jsonplaceholder.typicode.com/comments?postId=${start}&postId=${end}`)
    .then(response => response.json())
    .then(posts => { posts.forEach(add_post); });
}

function add_post(comment) {
  const post = document.createElement('div');
  post.className = 'post';
  post.innerHTML = comment.body;
  document.querySelector('#posts').append(post);
}
```
> 흐름: **스크롤 끝 감지(CH11) → fetch로 비동기 로드(CH13) → DOM에 추가**. 3챕터가 하나로 합쳐지는 대표 예제라 시험 가능성 높음.

# CH14. Fetch & APIs

> **한 줄**: `fetch`로 외부 **API**에서 데이터 가져오기. API = 소프트웨어용 인터페이스. CORS = 다른 출처 요청 정책.

## 1. 🔑 fetch 코드 스켈레톤 (checkStatus 포함) — 손코딩 1순위

```js
// Step 1. URL에서 데이터 fetch
function makeRequest() {
  fetch(url + possibleParameters)         // 일부 요청은 파라미터 필요
    .then(checkStatus)
    .then(response => response.json())     // JSON으로 올 때
    // .then(response => response.text())  // 텍스트로 올 때
    .then(processData)
    .catch(handleError);                   // 사용자 친화 에러 메시지 함수
}

// Step 2. 콜백 함수 구현
function checkStatus(response) {
  if (response.ok) {
    return response;
  } else {
    throw Error("Error in request");
  }
}

function processData(data) {
  // data로 뭔가 하기 (DOM 만들기 등)
}
```

## 2. 🔑 POST + FormData

- `fetch`는 **기본이 GET**. POST 하려면 옵션 객체 넘김.
```js
function makeRequest() {
  let url = "example.php";
  let myForm = document.getElementById('myForm');
  let params = new FormData(myForm);

  fetch(url, { method: "POST", body: params })
    .then(...)
    .then(...)
    .catch(...);
}
```

## 3. API (Application Programming Interface)

- UI(사용자용 인터페이스)와 비슷하지만 **소프트웨어용** (machine-readable vs human-readable).
- 같은 데이터/기능을 **다른 소프트웨어가 쉽게 소비(공유)** 하게 함. 앱끼리 대화.
- 소비하는 앱은 필요한 데이터/기능을 **서비스에 외주(outsource)** → 매번 "바퀴 재발명" 안 해도 됨 → 생산성↑.

## 4. Web API — 무엇을 어떻게 묻나

- API endpoint를 **HTTP**로 호출. **Request → Response** (JSON, XML, HTML, TXT 등).

| 묻는 것 | 설명 | 예 |
|---|---|---|
| **어디에** | 어느 API 주소 | `jsonplaceholder.typicode.com` |
| **권한** | key/token 필요? | jsonplaceholder는 X, exchangeratesapi는 O |
| **무엇을** | Route param vs Query param | 아래 ↓ |
| **어떤 포맷** | JSON/XML/HTML/text | — |

**Route parameters**: 값들이 **순서**대로, `/`로 구분
`http://numbersapi.com/154/math`

**Query parameters**: 명시적 **key/value** 쌍, `?`로 시작, `&`로 결합
`https://jsonplaceholder.typicode.com/comments?postId=1`

**예시 API**: NASA APOD(key 필요), Datamuse(key 불필요), XKCD(key 불필요·만화)

## 5. CORS (Cross-Origin Resource Sharing) ★

- 브라우저가 **다른 출처(origin)** 리소스 요청을 제한하는 정책 두 가지:
  - **SOP (Same-Origin Policy)**: **같은 출처**로의 요청만 허용
  - **CORS**: **다른 출처**의 리소스 요청을 **허용**하는 정책
- 즉 CORS는 한 도메인의 웹페이지가 **다른 origin의 리소스에 접근**하게 해주는 보안 메커니즘.
- 브라우저는 기본적으로 다른 사이트 리소스 접근을 **막음**(허가 없으면). 많은 API가 CORS 에러를 냄.
- **`Access-Control-Allow-Origin: *`**: 응답 헤더. **모든 origin의 접근 허용** (공개 API). 보통 더 잠겨있음.

# CH16. Node.js & Express

> **한 줄**: **서버에서** JS를 돌리는 런타임(Node) + 웹 프레임워크(Express)로 **API/서버**를 만든다.

## 1. 서버 사이드 & Node.js

- 지금까지: 클라이언트 스크립트 = **브라우저**에서 실행.
- 서버 코드: **서버에서 상시 실행** → 요청 대기 → 응답.
- **Node.js**: JS 실행 **런타임 환경**. 브라우저와 같은 **V8 엔진** 사용.
  - HTTP 요청, 파일 I/O, DB 등 **브라우저 없는 환경**용 API 제공.
  - **DOM/브라우저 객체 없음** → `document`, `window` 접근 불가. (★시험 함정)

## 2. 프로젝트 시작 순서 ★

```
1) 프로젝트 폴더 생성 (예: node-lab5)
2) npm init              → package.json (설정파일) 생성 (Enter로 기본값)
3) npm install express   → 모듈 설치
4) app.js 작성
5) public 폴더에 프론트 파일 (index.html, styles.css, ...)
6) node app.js           → 서버 시작
```

- 모듈 가져오기 = **`require()`** (파이썬 import 비슷). `const express = require("express");`
- **Express** = Node.js 위에서 도는 **웹 프레임워크** (JS로 작성됨).

## 3. 🔑 Express 앱 기본 골격 (손코딩 1순위)

```js
// 1. 모듈 불러오기 (let 말고 const — 실수로 덮어쓰기 방지)
const express = require('express');
// 2. 앱 인스턴스 생성
const app = express();
// 3. 라우트(엔드포인트) 추가
app.get("/", (req, res) => {
  res.send("Hello World");
});
app.get("/posts", function (req, res) {
  res.type("text").send("Visiting the posts route");
});
// 4. 포트 지정하고 서버 시작
const PORT = process.env.PORT || 8000;
app.listen(PORT);
```

- `'/'` = 루트 경로 (`http://localhost:8000/`).
- 라우트는 **정의된 순서대로** 매칭. 매칭 없으면 → **404 (Not Found)**.

## 4. 라우트 & req / res

```js
app.get(path, (req, res) => { ... });
```
- 2개 인자: **엔드포인트 URL 경로** + **콜백 함수**.
- **`req`** (HTTP 요청 객체): query, body, headers 보유.
- **`res`** (응답 객체): 클라에게 보낼 것.

**req 주요 속성**:
| 속성 | 설명 |
|---|---|
| `req.params` | 경로(path) 파라미터 |
| `req.query` | 쿼리 스트링 |
| `req.body` | 요청 본문 데이터 (**POST용**) |
| `req.cookies` | 쿠키 |

**res 주요 메서드** ★:
| 메서드 | 설명 |
|---|---|
| `res.send()` | 문자열 응답 + 연결 종료 (text/html) |
| `res.json()` | JSON 응답 (application/json) |
| `res.set()` | 헤더 설정 (`'Content-Type'`, `'text/html'`) |
| `res.type()` | content type 단축 ("text"→text/plain, "json"→application/json) |
| `res.status()` | 상태코드 설정 (`res.status(404).send('File not found')`) |
| `res.end()` | 빈 응답 (본문 없이) |

## 5. 요청 파라미터: Path vs Query ★

**Path/Named 파라미터**: `:파라미터명` 으로 정의
```js
// 경로: /CityWiki/states/:state/cities/:city
// URL:  http://localhost:8000/CityWiki/states/CA/cities/Irvine
// req.params: { state: "CA", city: "Irvine" }
app.get("/CityWiki/states/:state/cities/:city", function (req, res) {
  // req.params.city, req.params.state 로 접근
});
```

**Query String**: `req.query` 객체 (선택적 파라미터에 유용)
```js
// URL: http://localhost:8000/CityWiki?state=CA&city=Irvine
// req.query: { state: "CA", city: "Irvine" }
app.get("/CityWiki", function (req, res) {
  let state = req.query.state;  // CA
  let city = req.query.city;    // Irvine
});
```
- 쿼리스트링은 경로에 안 포함됨 → 특정 key 존재를 확신 못 함 → **없으면 에러 보내야 함**.

## 6. 에러 코드 ★

| 코드 | 의미 | 언제 |
|---|---|---|
| **400** Bad Request | 클라이언트 잘못 | 잘못된 파라미터 형식, 필수 쿼리 누락 |
| **500** Internal Server Error | 서버 잘못 | 요청 파라미터와 무관, SQL 연결 에러 등 |

```js
if (!(state && city)) {
  res.status(400).send("Error: city and state required.");
}
```

## 7. 정적 파일 서빙 (static assets)

```js
app.use(express.static('public'));   // public 폴더를 루트로 노출
```
- `public/index.html` → 루트 도메인에서 서빙. `public/images/dog.jpg` → `/images/dog.jpg`.
- 여러 번 호출 가능 (`express.static("public")`, `express.static("media")`).

## 8. 🔑 POST 요청 처리 (미들웨어) ★★

- GET은 URL로(`req.params`/`req.query`), **POST는 본문(body)으로** 파라미터 전송 → **`req.body`** 사용.
- POST 데이터 타입 3종: `x-www-form-urlencoded`, `application/json`, `multipart/form-data`.
  - 앞 2개는 Express 기본 지원. `multipart/form-data`(FormData 사용 시)는 **`multer`** 모듈 필요.

```js
const multer = require("multer");

app.use(express.urlencoded({ extended: true }));  // x-www-form-urlencoded
app.use(express.json());                          // application/json
app.use(multer().none());                         // multipart/form-data

app.post("/signup", (req, res) => {
  let name = req.body.name;     // params/query 아닌 body!
});
```
- POST를 URL로 보내면 단점: **안전하지 않고, 데이터 길이 제한**.

## 9. 🔑 파일 I/O (fs 모듈) ★★

- `fs` = file system 모듈. **core 모듈이라 npm install 불필요**.

**콜백 버전**:
```js
const fs = require('fs');
fs.readFile('example.txt', 'utf8', (err, data) => {
  if (err) { handleError(err); }
  else { processData(data); }   // data = 파일 내용
});
```

**Promise 버전** (`require('fs').promises`):
```js
const fs = require('fs').promises;
// then/catch
let data = fs.readFile('example.txt', 'utf8');
data.then(result => console.log("Hello!" + result)).catch(handleError);
// async/await
let data2 = await fs.readFile('example.txt', 'utf8');
```

**async 핸들러** (await는 async 함수 안에서만):
```js
app.get('/posts', async (req, res) => {
  try {
    await fs.readFile('posts.txt', 'utf8');
    res.type('text').send('Files read successfully!');
  } catch (err) {
    res.status(500).type('text').send('Some error in file reading!');
  }
});
```

**쓰기**:
```js
fs.writeFile('example.txt', 'Sample Text', (err) => { ... });  // 있으면 덮어씀
await fs.writeFile('posts.txt', post + '\n', { flag: 'a' });   // append
// await fs.appendFile('posts.txt', post + '\n');              // append 다른 방법
```

**파일 없음 에러**: `error.code === 'ENOENT'` (Error No Entry, 요청했는데 없음).

**JSON 읽기/쓰기**: `JSON.parse(jsonString)`, `JSON.stringify(jsonObject)`
```js
let products = await fs.readFile('products.json', 'utf8');
let data = JSON.parse(products);
data['type'] = 'TV';
await fs.writeFile('products.json', JSON.stringify(data));
```

# CH17. SQL ⭐⭐⭐ (양 많고 시험 비중 큼 — 표 위주 암기)

> **한 줄**: 관계형 DB에서 데이터를 **저장·조회·수정·삭제**하는 표준 언어. SELECT/INSERT/UPDATE/DELETE/JOIN.

## 1. 풀스택 & 관계형 DB 기초

- 풀스택 3계층: **브라우저(HTML/CSS/JS)** ─ **웹서버(Node.js)** ─ **DB서버(SQL)**.
- 관계형 DB: 데이터를 **테이블**(행 rows + 열 columns)에 저장. DBMS가 저장·관리.

## 2. Primary Key & Foreign Key ★★ (무결성 규칙)

| | **Primary Key (기본키)** | **Foreign Key (외래키)** |
|---|---|---|
| 정의 | 테이블의 **행을 고유 식별**하는 열(또는 열 집합) | 다른 테이블의 **PK를 참조**하는 열 |
| 규칙 | 모든 행에 값 존재 + 고유 → **개체 무결성(Entity Integrity)** | 모든 FK 값은 다른 테이블 PK에 존재해야 → **참조 무결성(Referential Integrity)** |
| 역할 | 이 테이블 데이터 참조용 | 여러 테이블 데이터를 **JOIN**으로 연결 |

- **Composite Primary Key (복합 기본키)**: 여러 열을 묶어 PK (예: AuthorISBN의 `isbn`+`authorID` 조합이 unique).
- 관계: **1:N (one-to-many, ∞ 기호)**, **N:M (many-to-many)**.

**자주 나는 에러 3종**:
1. PK 열에 값 안 줌 → 개체 무결성 위반.
2. 여러 행에 같은 PK 값 → 에러.
3. 다른 테이블 PK에 없는 FK 값 → 참조 무결성 위반.

## 3. 🔑 CREATE TABLE + Foreign Key (손코딩 가능)

```sql
CREATE TABLE departments (
  dept_id int autoincrement primary key,
  dept_name varchar(45) not null
);

CREATE TABLE employees (
  emp_id int autoincrement primary key,
  emp_name varchar(55) not null,
  dept_id int,
  FOREIGN KEY (dept_id) REFERENCES departments (dept_id)
);
```
→ employees → departments 로 **N:1** 관계 생성.

**복합키 + 다중 FK** (many-to-many 연결 테이블):
```sql
CREATE TABLE AuthorISBN (
  isbn text,
  authorID int,
  PRIMARY KEY (isbn, authorID),
  FOREIGN KEY (isbn) REFERENCES Titles (isbn),
  FOREIGN KEY (authorID) REFERENCES Authors (authorID)
);
```

## 4. 데이터 타입 ★ (SQL vs SQLite 변환 — 시험 함정)

| SQL | 설명 | → SQLite |
|---|---|---|
| `INT` | 정수 | `INTEGER` |
| `VARCHAR(n)` | 최대 n글자 문자열 | `TEXT` |
| `TINYINT` | 아주 작은 수 (보통 0~255) | `INTEGER` |
| `DATETIME` | 날짜+시간 `YYYY-MM-DD HH:MM:SS` | `NUMERIC` |
| `DOUBLE(size, d)` | 부동소수 | `REAL` |
| `TEXT` | 큰 문자열 | `TEXT` |

**SQLite 타입은 4개만**: `INTEGER`, `REAL`, `TEXT`, `BLOB`(이진 대용량).
- **SQLite엔 Boolean 타입 없음** → 정수 **0(false) / 1(true)** 로 저장. (★시험)

## 5. SQL 키워드 & 대소문자 규칙 ★

- **SQL 키워드는 대소문자 구분 안 함** (SELECT = select).
- **테이블명·열명은 대소문자 구분함**. (★함정)
- 핵심 키워드: `SELECT`, `FROM`, `WHERE`, `INNER JOIN`, `GROUP BY`.

## 6. 🔑 SELECT (조회) — 손코딩 필수

```sql
SELECT * FROM tableName;                    -- 모든 열
SELECT authorID, lastName FROM Authors;     -- 특정 열
SELECT DISTINCT City FROM Customers;        -- 중복 제거
```
- 결과 = **result set / result table**. 주석: `--` (한 줄), `/* */` (여러 줄).

## 7. 🔑 WHERE (조건) ★★

```sql
SELECT col1, col2 FROM tableName WHERE criteria;
SELECT title, editionNumber, copyright FROM Titles WHERE copyright > 1999;
```
> 쿼리 쓸 때 생각 순서: **FROM → WHERE → SELECT** 순으로 떠올려.

**WHERE 연산자**:
| 연산자 | 의미 |
|---|---|
| `=` `>` `<` `>=` `<=` | 비교 |
| `<>` | 같지 않음 (일부 SQL은 `!=`) |
| `BETWEEN` | 범위 |
| `LIKE` | 패턴 매칭 |
| `IN` | 여러 값 중 하나 |

**LIKE 와일드카드** ★:
- `%` = **0개 이상**의 아무 문자, `_` = **정확히 1개** 문자.
```sql
SELECT ... FROM Authors WHERE lastName LIKE 'D%';   -- D로 시작
SELECT ... FROM Authors WHERE lastName LIKE '_i%';  -- 2번째 글자가 i
```
- **문자열은 작은따옴표(')** 로 감쌈. **숫자는 따옴표 X**.
- `AND` / `OR` / `NOT` 로 조건 결합.

## 8. 🔑 ORDER BY (정렬)

```sql
SELECT col1, col2 FROM tableName ORDER BY column ASC;
SELECT col1, col2 FROM tableName WHERE 조건 ORDER BY column DESC;
```
- **ASC = 오름차순(기본)**, **DESC = 내림차순**.

## 9. 집계 함수

`MIN()` 최소 · `MAX()` 최대 · `COUNT()` 행 개수 · `AVG()` 평균 · `SUM()` 합계.

## 10. 🔑 INSERT / UPDATE / DELETE ★★ (손코딩 1순위)

**INSERT** (행 추가):
```sql
INSERT INTO Authors (firstName, lastName) VALUES ('Sue', 'Smith');
```
- auto-increment 열(authorID)은 **값 지정 안 함** (자동 부여). 모든 열에 값 넣으면 열 이름 생략 가능(순서 주의).
- 문자열 안 작은따옴표는 **두 개로 escape**: `'O''Malley'`.

**UPDATE** (수정):
```sql
UPDATE Authors SET lastName = 'Jones'
WHERE lastName = 'Smith' AND firstName = 'Sue';
```
- ⚠️ **WHERE 빼면 모든 행이 수정됨!**

**DELETE** (삭제):
```sql
DELETE FROM Authors WHERE lastName = 'Jones' AND firstName = 'Sue';
```
- ⚠️ **WHERE 빼면 모든 행 삭제!**

## 11. NULL 처리 ★

- NULL = **값 없는 필드**. 선택적 필드는 값 없이 저장 가능 → NULL.
- **`=`, `<`, `<>` 로 NULL 테스트 불가** → **`IS NULL` / `IS NOT NULL`** 사용. (★시험)
```sql
SELECT CustomerName, Address FROM Customers WHERE Address IS NULL;
```

## 12. 🔑 INNER JOIN ★★★ (시험 핵심)

- 두 테이블에서 **매칭되는 값이 있는** 행만 선택.
```sql
SELECT col1, col2, ...
FROM table1 INNER JOIN table2
ON table1.columnName = table2.columnName;
```
- **`ON`**: 어느 열끼리 비교해 결합할지 명시.

**예 (저자 ↔ ISBN)**:
```sql
SELECT firstName, lastName, isbn
FROM Authors INNER JOIN AuthorISBN
ON Authors.authorID = AuthorISBN.authorID
ORDER BY lastName, firstName;
```

**4개 테이블 조인** (중첩):
```sql
SELECT Titles.title, Titles.isbn, Authors.firstName,
       Authors.lastName, Titles.copyright, Publishers.publisherName
FROM
  ( Publishers INNER JOIN Titles
    ON Publishers.publisherID = Titles.publisherID )
  INNER JOIN
  ( Authors INNER JOIN AuthorISBN
    ON Authors.authorID = AuthorISBN.authorID )
  ON Titles.isbn = AuthorISBN.isbn
ORDER BY Titles.title;
```
- ⚠️ 여러 테이블에 **같은 이름 열**이 있으면 **`테이블명.열명`** 으로 명시 (예: `Authors.authorID`). 안 하면 에러.

# CH18. SQLite in Node.js

> **한 줄**: Node에서 파일 I/O 대신 **SQLite DB**를 써서 데이터 조회·삽입. `db.all`(조회), `db.run`(실행), `?` 플레이스홀더(SQL injection 방어), `db.close`.

## 1. 왜 파일 I/O 대신 DB? ★

- 파일 I/O 한계: **지루·비효율·오류 잦음**. "5월 25일 글만", "특정 단어 포함 글만" 같은 조건 검색이 **비싼 반복문(expensive looping)** 필요.
- **DBMS 장점**: 대량 구조화 데이터 처리, **빠른 검색**, 대용량 확장, **장애 복구**, **다중 사용자** 동시 지원.

## 2. 모듈 설치 & 차이 ★

```
npm install sqlite3 sqlite
```
| 모듈 | 특징 |
|---|---|
| `sqlite3` | SQLite 연결용 가장 인기. 결과/에러를 **콜백**으로 |
| `sqlite` | `sqlite3`을 감싼 **wrapper, Promise 추가** |

## 3. 🔑 DB 연결 함수 (손코딩 1순위) ★★

```js
const sqlite3 = require('sqlite3');
const sqlite = require('sqlite');

// app.js 어딘가 (엔드포인트 정의들 뒤에)
async function getDBConnection() {
  const db = await sqlite.open({
    filename: '<db-name>.db',
    driver: sqlite3.Database
  });
  return db;
}
```

## 4. 🔑 db.all() — 조회 (결과 배열 반환) ★★

```js
let query = "SELECT authorID, lastName FROM Authors LIMIT 2;";
let db = await getDBConnection();
let rows = await db.all(query);
console.log(rows);
// => [ { authorID: 1, lastName: 'Deitel' }, { authorID: 2, lastName: 'Deitel' } ]
```
- 결과는 **객체 배열(result set)**. 열 이름으로 **점 표기(dot notation)** 접근:
```js
let first = rows[0];
console.log("Author Name: " + first.lastName + "(" + first.authorID + ")");
// => Author Name: Deitel(1)
```
- ⚠️ **SELECT에 명시한 열만** 접근 가능 (위 예는 authorID, lastName만).

## 5. 에러 핸들링

```js
try {
  let rows = await db.all("SELECT lastName FROM Authors;");
  // 결과 처리
} catch (error) {
  res.status(500).send("Error on the server. Please try again later");
}
```

## 6. 🔑 SQL Injection & `?` 플레이스홀더 ★★★ (시험 단골)

**문제** — 문자열 결합은 위험:
```js
let query = "SELECT * FROM users WHERE username='" + user +
            "' AND password='" + pswd + "'";
```
- 해커가 `user`에 **`' OR 1=1 --`** 넣으면 → 조건이 항상 참 → **데이터 유출**.
  (`-- ` 뒤는 주석 처리되어 password 검사 무력화)

**해결** — `?` 플레이스홀더 (문자열 결합 ❌, 별도 매개변수로 대입):
```js
let query = "SELECT * FROM users WHERE username=? AND password=?";
let result = await db.all(query, [user, pswd]);  // 배열 순서 = ? 순서
```
- ✅ 쿼리가 **내부적으로 escape** 되어 안전. `?` 순서 = 배열 인덱스 순서.

## 7. 🔑 db.run() — INSERT/UPDATE/DELETE (반환값 없음)

```js
let db = await getDBConnection();
let sqlString = 'INSERT INTO Authors (firstName, lastName) VALUES (?, ?)';
await db.run(sqlString, ['Sue', 'Smith']);
```
- `db.run(sqlString, paramsArray)`: 단일 SQL 실행 + `?`에 배열 값 대입. **반환값 없음**.

## 8. 🔑 db.close() — 연결 닫기 ★

```js
let db = await getDBConnection();
let result = await db.all(query);
await db.close();   // 안 닫으면 memory leak!
```
- 작업 끝나면 **항상 close** → 자원 해제, 메모리 누수 방지.

## 🎯 개념 함정 빠른 체크 (틀리기 쉬운 것)
- mouse `button`(0좌1중2우) vs `which`(0없1좌2중3우) — **숫자 다름**.
- 키 이벤트 순서 `keydown→keypress→keyup`, 모든 키 감지는 **keydown**.
- Promise 상태 3개: pending / fulfilled(→then) / rejected(→catch).
- async/await 에러는 **try/catch** (not .then/.catch).
- `Promise.all` = 전부 성공해야 / `Promise.any` = 하나만 성공하면.
- Node엔 **document·window 없음**.
- POST는 **req.body** (req.params/query 아님).
- 400=클라 잘못 / 500=서버 잘못.
- SQL 키워드는 대소문자 무시, **테이블·열명은 구분**.
- LIKE: `%`=여러문자, `_`=한문자. 문자열은 작은따옴표.
- NULL은 `=` 못 씀 → **IS NULL / IS NOT NULL**.

- `db.all`=조회(배열반환) / `db.run`=실행(반환없음).
- 연결은 항상 **db.close()**.
